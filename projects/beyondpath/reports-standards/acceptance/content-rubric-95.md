# BeyondPath 報告 · 95 分內容深度 Rubric

> **建立日期**：2026-04-21
> **作者**：🧙 霍爾（CPO / 策略 · Opus 4.7）
> **目的**：為 5 份報告（PATH 診斷 / Lab 分析 / 市場探測 / 市場報告 / AI 問卷）建立**客觀可驗收**的內容深度評分標準
> **用法**：作為 Layer 1 Content Spec 與 Layer 3 Prompt Rubric 的「驗收側標準」；Gate 3 遠景校準時逐條檢查
> **底線**：全 15 格平均 ≥ 9.5，任一格 < 9 不得進 Gate 1

---

## 零、三層評分框架（Surface / Mid / Deep）

三層對應三種讀者需求：

| 層 | 英文 | 回答什麼 | 讀者需求 |
|---|------|---------|---------|
| **Surface** | What | 數字是多少、趨勢長怎樣、誰是 top | 快速掃一眼帶走數字 |
| **Mid** | Why | 為什麼這個數字代表 X 意義、脈絡是什麼、和同業比如何 | 能跟老闆／投資人說故事 |
| **Deep** | Now what | 接下來做什麼、誰做、何時做、風險是什麼、怎麼驗收 | 關掉報告就能動手 |

**評分刻度**（每格 10 分制，0.5 為最小單位）：

| 分數 | 定義 |
|------|------|
| **10** | Top-player 水準 · 可當對外提案材料 · 內容有品牌辨識度 |
| **9.5** | 95 分門檻 · 所有硬指標達標 · 可進 Gate 1 |
| **9** | 95 分前一階 · 缺 1-2 項軟指標（如語氣、比喻精準度）· 需微調 |
| **8.5** | MVP 尚可水準 · 缺結構性要素（如 Next Action 缺時程）· 不進 Gate |
| **≤ 8** | 不合格 · 退回 prompt 重跑 |

## 二、PATH 診斷

### §2.1 Surface · PATH 診斷數據展示

| 分數 | Criteria |
|------|----------|
| **10** | P/A/T/H 四軸 score 用**雙編碼**（數值 + 狀態色 + 文字標籤 healthy/warn/danger）· 有同業 benchmark 對比線（%ile 排名）· 每題分數可點擊展開原始作答 |
| **9.5** | P/A/T/H 四軸 score 雙編碼（數值 + 狀態色）· 每題分數可看小計 · 附同業中位數 baseline |
| **9** | P/A/T/H 四軸數值明確 · 狀態色正確 · 缺 benchmark 或作答原文溯源 |

**硬指標**：
- 總分 0-100 · 四軸各 0-25 · 整數呈現
- 狀態分級：≥ 80 healthy（teal）/ 60-79 warn（gold）/ < 60 danger（rose）
- Benchmark：同階段新創中位數（最少 AI 推估，標🟡；理想 DataForSEO 或公開資料庫）
- **禁止**：Math.random、純色分級（色盲不友善）、< 60 分用 primary purple（語義衝突）

**反例**：「你的 PATH 分數 68/100」——缺四軸拆解、缺 benchmark、缺狀態語義

### §2.2 Mid · PATH 診斷原因解釋

| 分數 | Criteria |
|------|----------|
| **10** | 每軸**一句 diagnosis**（為什麼是這個分數）+ 引用用戶作答原文 2-3 句當證據 + 同業對照（「你 A 軸 18/25，同價位產品中位數 14」）+ **風險警示**（若不補強 Execution，3 個月後 ____） |
| **9.5** | 每軸一句 diagnosis + 原文證據 1-2 句 + 同業對照數字（推估可） |
| **9** | 每軸一句 diagnosis + 原文證據 · 缺同業對照 |

**硬指標**：
- Diagnosis 字數：每軸 30-60 字（不含證據）
- 原文證據：至少 1 句用戶作答直引（用引號標出 · `"我不知道怎麼跟客戶開口收錢"`）
- 同業對照必附**資料來源 badge**（🟢實測 / 🟡推估 / 🔴AI 生成）
- **禁止**：通用雞湯（「加油！你還可以更好」）、只講分數不講語義

**反例**：「Problem 18/25，表現不錯」——沒說明為什麼不錯、和誰比不錯、哪題拉分哪題扣分

### §2.3 Deep · PATH 診斷 Next Action

| 分數 | Criteria |
|------|----------|
| **10** | **Top 3 actions this 2 weeks**（對標 Productboard 格式）· 每條含「誰做 / 工時 / 成功指標 / 風險 / 失效條件」五要素 · **跨工具導引**（「Audience 模糊 → 跑 Lab 研究」）· 預期成績「若全部完成，PATH 分數預估 +8」 |
| **9.5** | Top 3 actions · 每條含「誰做 / 工時 / 成功指標」三要素 · 至少 1 條 cross-tool 導引 |
| **9** | Top 3 actions · 每條含「工時 + 成功指標」· 缺「誰做」或缺跨工具 |

**硬指標**：
- Action 數量：**≥ 3 條**（不超過 5 條，避免 decision paralysis）
- 每條格式：
  ```
  Action 1 · 補 Traction 證據
  誰做：你 + 一位實習生
  工時：預估 6 小時 · 移動城堡 1 小時
  成功指標：蒐齊 3 個付費客戶的 testimonial video（每段 ≥ 60 秒）
  風險：客戶可能不想露臉 → 備案改文字 + logo
  失效條件：若找不到任何付費客戶 → 先補驗證層（跑問卷引擎）
  ```
- **工時雙軌強制**：預估工時 / 移動城堡工時（憲法 v3.4）
- **禁止**：「加強 marketing」這類無法執行的模糊建議

**反例**：「建議補強 Traction 與 Execution」——沒告訴讀者「補哪個、先補哪個、怎麼補、補完怎麼判斷成功」

## 三、Lab 分析

### §3.1 Surface · Lab 分析數據展示

| 分數 | Criteria |
|------|----------|
| **10** | Persona 卡**強制標記合成來源**（🔴 AI 合成 基於 N 份資料點 / 🟢 真人訪談 N 位）· Quote 直引區（對標 Dovetail）· 情緒分佈圖（正/中/負比例）· 關鍵詞雲（topic cluster · 對標 Dovetail Magic Cluster） |
| **9.5** | Persona 卡標合成來源 · Quote 直引 3-5 則 · 情緒分佈 · 關鍵詞 top 10 |
| **9** | Persona 卡標來源 · Quote 2-3 則 · 缺情緒分佈或 cluster |

**硬指標**：
- Persona 卡必含：**來源標記（🔴/🟢）+ 基於幾份資料點 + 合成信心度 %**
- Quote 格式：`"原文" — 來源（P3 · 45 歲 · SaaS 工程師）`
- 情緒分佈：正 / 中 / 負 百分比 + 總樣本數 N
- **禁止**：無標記的 AI 合成 persona（P3 坤書必炸）、用 primary purple 標負面情緒（語義衝突 · 憲法）

**反例**：「根據用戶訪談，大部分用戶覺得好用」——沒量化、沒原文、沒來源

### §3.2 Mid · Lab 分析原因解釋

| 分數 | Criteria |
|------|----------|
| **10** | **Insight clustering**（對標 Dovetail · 10 條原始 insight → 聚合成 3 組主題）· 每組主題附「支撐強度」（幾則 quote 支撐）+ 主題間對比（「P1 偏好 X，P3 偏好 Y，衝突點在 ____」）+ **JTBD 翻譯**（「用戶雇用你來解決什麼問題」→ 用生活化比喻） |
| **9.5** | Insight 聚合成 2-3 組主題 · 每組支撐強度標示 · 有 JTBD 翻譯 |
| **9** | Insight 有聚合 · 缺主題對比 |

**硬指標**：
- **Insight cluster 定義**（對標 Dovetail）：
  - 原始 insight ≥ 6 條才需聚合
  - 聚合後 3-5 組（不超過 5，避免失去摘要意義）
  - 每組 label 為動詞開頭句（「想要快速看到成效」而非「效率」）
  - 每組支撐：**≥ 2 則 quote 支撐才算成立**
- **JTBD 翻譯表強制**（見 §7 術語表）
- **禁止**：只列 10 條 insight 不聚合、cluster label 用抽象名詞（「滿意度」）

**反例**：「insight 1: 用戶覺得慢 / insight 2: 用戶想要快 / insight 3: 用戶不滿」——三條其實是同一組，沒聚合

### §3.3 Deep · Lab 分析 Next Action

| 分數 | Criteria |
|------|----------|
| **10** | **3 件立即可改**（Landing headline / persona 卡補完 / 問卷追加題）+ 每條 wireframe-level 具體建議（「改成『1 小時內看到第一份報告』而非『快速產出』」）+ 追蹤機制（「改完 2 週後追蹤 bounce rate」）+ 追加研究方向（「本次樣本偏 P1，建議下輪跑 P3」） |
| **9.5** | 3 件具體可改 · 每條有 before/after 對照 · 有追蹤機制 |
| **9** | 3 件具體可改 · 缺 before/after 或追蹤 |

**硬指標**：
- Action 數量：**≥ 3 條**
- 每條含**文字層級的 before/after**（不是只說「改文案」而是給出建議新版）
- 追蹤機制含具體 KPI（bounce rate / conversion / retention），不是「觀察一下」
- **研究迭代建議**：至少 1 條「下輪 Lab 該補誰」
- **禁止**：「建議優化 landing」這類無法執行

**反例**：「建議改善 onboarding 流程」——沒說改哪句、改成什麼、怎麼驗收

## 四、市場探測

### §4.1 Surface · 市場探測數據展示

| 分數 | Criteria |
|------|----------|
| **10** | **搜尋量 / CPC / 競爭強度三軸**（每軸附 sparkline 12 個月）· 資料源三色 badge（DataForSEO 🟢 / AI 推估 🟡 / AI 生成 🔴）· Top 5 相關關鍵字（含搜尋量排序）· 地區分佈熱圖（台灣縣市層級） |
| **9.5** | 搜尋量 / CPC / 競爭三軸 · sparkline 或趨勢圖 · 資料源 badge · Top 5 關鍵字 |
| **9** | 三軸數據 · 有資料源 badge · 缺 sparkline 或地區分佈 |

**硬指標**：
- **Math.random 絕對零容忍**（P3 坤書打開 DevTools 必炸 · 憲法級）
- 每個數字必有**資料源 badge**（🟢/🟡/🔴）· badge 可點擊看方法論
- Sparkline 顯示**至少 6 個月**趨勢
- **禁止**：用 Tailwind blue #3B82F6（憲法違規）、未標記的 AI 推估數字

**反例**：「市場搜尋量 12,000」——沒標來源、沒趨勢、沒同類比較

### §4.2 Mid · 市場探測原因解釋

| 分數 | Criteria |
|------|----------|
| **10** | **一句判讀**（「訊號強 / 擁擠但有 niche / 已紅海」）+ 判讀的 3 個支撐理由 + 相鄰市場對比（「跟 ____ 比，你這個 niche 窄一半但 CPC 低 3 倍」）+ **紅藍海象限定位**（視覺圖 + 文字說明）|
| **9.5** | 一句判讀 + 3 個支撐 + 相鄰市場比較（至少 1 個） |
| **9** | 一句判讀 + 2-3 個支撐 · 缺相鄰市場 |

**硬指標**：
- 判讀句 20-40 字 · 以名詞 + 形容詞開頭（「市場擁擠但 niche 可切」）
- 支撐理由**引用具體數字**（「CPC NT$45 低於產業中位數 NT$80」）
- 紅藍海象限軸線：X = 市場大小 / Y = 競爭密度
- **禁止**：「市場前景看好」這類無依據判讀、用「藍海」「紅海」卻不定義量化標準

**反例**：「這個市場競爭激烈」——沒定義什麼叫激烈、和誰比、數字在哪

### §4.3 Deep · 市場探測 Next Action

| 分數 | Criteria |
|------|----------|
| **10** | **3 條驗證動作**（跑問卷 / 聯絡 N 位目標客戶 / 跑 PATH 診斷）· 每條含「時程 / 成本 / 期望結論」· **Go/No-Go 判準**（「若 3 週內收到 ≥ 10 張意向表 → Go，反之 → 回頭調定位」）· **相鄰切入路徑**（若本 niche 不行，備案往哪走）|
| **9.5** | 3 條驗證動作 · 有時程 / 成本 · 有 Go/No-Go 判準 |
| **9** | 3 條動作 · 有時程 · 缺 Go/No-Go |

**硬指標**：
- Action 數量：**≥ 3 條**
- 每條含：時程（週）/ 預算（NT$）/ 期望樣本量
- **Go/No-Go 判準必含數值門檻**（不是「反應熱烈」而是「≥ N 張表單」）
- **禁止**：「繼續觀察市場」這類無行動項

**反例**：「建議進一步驗證市場」——沒告訴讀者驗什麼、驗多久、怎麼判斷驗證成功

## 五、市場報告（完整分析版）

> **與市場探測的差異**：探測 = 訊號層（20 分鐘讀完）· 報告 = 完整分析層（可當投資人提案材料 · 30-60 分鐘）

### §5.1 Surface · 市場報告數據展示

| 分數 | Criteria |
|------|----------|
| **10** | TAM/SAM/SOM 三層 funnel 圖（含計算式展開）· 競品 landscape 矩陣（2×2 X=定位 Y=成熟度）· 成長驅動因子 top 5 · 監管/合規風險清單 · **時間切片**（近 3 年趨勢 + 未來 18 個月預測） |
| **9.5** | TAM/SAM/SOM + 競品矩陣 + 成長驅動 + 3 年趨勢 |
| **9** | TAM/SAM/SOM + 競品矩陣 · 缺時間切片或風險清單 |

**硬指標**：
- TAM/SAM/SOM **計算式必明示**（「TAM = 台灣 1-20 人團隊數 × 人均年軟體預算」= 40 萬團隊 × NT$36,000 = NT$144 億）
- 計算式每項變數附**資料來源 badge**
- 競品矩陣至少含 8-12 個對手（台灣 / 亞洲 / 國際三層）
- **禁止**：TAM 直接報數字不解釋怎麼算（投資人立刻 reject）

### §5.2 Mid · 市場報告原因解釋

| 分數 | Criteria |
|------|----------|
| **10** | **Thesis 統領句**（一句話定義投資假設）· 3 支撐柱（demand / supply / timing）· 每柱引用至少 2 個 third-party 來源 · **反方論點公平呈現**（「但 ____ 分析師認為 ____」）+ 反駁 · 情境分析（樂觀 / 基線 / 悲觀三軸） |
| **9.5** | Thesis + 3 支撐柱 + 反方論點 + 情境分析（至少 2 軸） |
| **9** | Thesis + 3 支撐柱 · 缺反方或情境 |

**硬指標**：
- Thesis 30-50 字 · 必含時間窗（「未來 18 個月」）
- 每支撐柱 ≥ 2 third-party 來源（Gartner / IDC / CB Insights / McKinsey / 學術論文）
- **反方論點是 95 分門檻的差異化**——只報正面不夠，必須公平呈現反方（McKinsey POV 標準）
- 情境分析：樂觀 / 基線 / 悲觀各自機率 %
- **禁止**：只講利多不講利空、single source 就下結論

### §5.3 Deep · 市場報告 Next Action

| 分數 | Criteria |
|------|----------|
| **10** | **分三時相 Roadmap**（0-3 月 / 3-12 月 / 12+ 月）· 每時相 3-5 個 action · 每 action 含 KPI + 負責角色 + 預算帶 · **Exit scenarios**（若 X 指標達成 → 進 A 階段 / 若 Y 指標未達 → 回到驗證層）· 對外投資人版 vs 對內執行版**雙敘事** |
| **9.5** | 三時相 Roadmap + 每時相 3 action + Exit scenarios |
| **9** | 三時相 Roadmap · 缺 Exit scenarios 或雙敘事 |

**硬指標**：
- Roadmap 視覺：timeline chart（不是純文字）
- Exit scenarios ≥ 3 個分支
- **雙敘事差異**（對外/對內）：
  - 對外版：去執行細節、保留 thesis + 3 里程碑 + 需要資源
  - 對內版：含每週 checklist、人員分配、預算明細
- **禁止**：把對外 pitch 直接當對內 playbook 用

## 六、AI 問卷

### §6.1 Surface · AI 問卷數據展示

| 分數 | Criteria |
|------|----------|
| **10** | 題型配比圖（單選 / 多選 / 開放 / Likert / NPS 比例）· 每題**預期完成率**（基於題型 + 位置 AI 推估）· **認知負擔分數**（每題 0-10，> 7 紅字警告）· 估計填答時間 + 放棄率預測 |
| **9.5** | 題型配比 + 預期完成率 + 估計填答時間 |
| **9** | 題型配比 + 填答時間 · 缺認知負擔 |

**硬指標**：
- 每題附**預期完成率 %**（AI 推估 🟡 · 基於題型歷史資料）
- 估計填答時間用**範圍**（「3-5 分鐘」而非「3 分鐘」）· 避免用戶期望值錯誤
- 放棄率預測分節呈現（第 N 題後預期放棄 X%）
- **禁止**：僅列題數不談 UX 預測（問卷設計的意義就在此）

### §6.2 Mid · AI 問卷原因解釋

| 分數 | Criteria |
|------|----------|
| **10** | **每題設計意圖**（「這題為了驗證 X 假設」）· 題目間**邏輯 DAG**（哪題 depends on 哪題）· **bias 自檢**（「本問卷有 ____ 傾向，建議在 Q5 加反向題」）· 對標學術問卷設計原則（Likert scale 是否平衡、leading question 檢查） |
| **9.5** | 每題設計意圖 + 邏輯關係 + bias 自檢 |
| **9** | 每題設計意圖 · 缺 bias 自檢 |

**硬指標**：
- 每題設計意圖 20-40 字
- Bias 自檢項目：leading question / double-barreled / 社會期許偏差 / 順序效應
- 邏輯 DAG 視覺化（node-edge graph 或文字 tree）
- **禁止**：只列問題不講理由（AI 問卷的差異化就是「解釋為何這樣設計」）

### §6.3 Deep · AI 問卷 Next Action

| 分數 | Criteria |
|------|----------|
| **10** | **發送前 checklist**（試填 ≥ 3 人 / 逐題 bias 檢查 / 預期樣本量計算）· **分發通路建議**（FB 社團 / 電子報 / 付費 panel，各自成本與代表性）· **分析計劃預告**（收到 N 份後用什麼方法分析 · 對接 Lab 研究報告格式）· **追加問卷觸發條件**（若樣本 < X → 第二波） |
| **9.5** | 發送前 checklist + 分發通路 + 分析計劃 |
| **9** | 發送前 checklist + 分發通路 · 缺分析計劃 |

**硬指標**：
- Checklist ≥ 5 項
- 分發通路 ≥ 3 個選項，每個含成本、代表性、預期樣本量
- 分析計劃必**預告對接到 Lab 研究報告的哪個 section**
- **跨工具導引強制**：指出「收到回覆後 → 走 Lab 分析報告」
- **禁止**：發送後就沒下文（問卷只是中繼站，不是終點）

## 七、術語翻譯表（台灣生活化比喻 · 5 份報告共用）

**規則**：報告中首次出現英文術語必附括號翻譯 · 全報告最多出現 1 次原文，後續全用中文。

| 英文 | 生活化翻譯 | 類比場景 |
|------|----------|---------|
| **Churn** | 客戶流失率 | 訂閱制手搖飲月退率 |
| **JTBD**（Jobs-to-be-Done）| 用戶雇用你解決什麼問題 | 買電鑽不是要電鑽，是要牆上的洞 |
| **PMF**（Product-Market Fit）| 產品和市場對上了沒 | 像相親——不只你喜歡對方，對方也要肯回訊息 |
| **CAC**（Customer Acquisition Cost）| 抓一個客戶要花多少錢 | 夜市擺攤的傳單 + 試吃成本 |
| **LTV**（Lifetime Value）| 這個客戶一輩子貢獻多少 | 熟客一年消費總額 |
| **TAM / SAM / SOM** | 整個市場 / 你能碰到的 / 你能拿到的 | 夜市 / 你這條街 / 你這攤的客人 |
| **NPS**（Net Promoter Score）| 你客戶推薦別人的意願 | 吃完會不會跟朋友說「你一定要去」 |
| **Onboarding** | 新手上路引導 | 新員工第一週的 OJT |
| **Retention** | 回購/留存率 | 髮廊回頭客比例 |
| **Conversion rate** | 轉換率 | 100 人進店多少人結帳 |
| **MRR / ARR** | 月經常性收入 / 年經常性收入 | 月費健身房穩定收入 |
| **Niche** | 小眾切入點 | 不做整個便當，專做低碳餐盒 |
| **Red ocean / Blue ocean** | 紅海 / 藍海 | 紅海 = 士林夜市，藍海 = 開在捷運尾站 |
| **Persona** | 用戶原型 | 像偶像劇人物設定 |
| **Hypothesis** | 我猜這樣，但要驗證 | 「這家店生意好應該是因為便宜」的那個「應該」 |
| **Signal** | 訊號 | 暗戀對象已讀不回 vs 秒回 |
| **Insight** | 洞察 | 發現阿嬤買菜只挑下午 4 點去，因為肉攤要收攤打折 |
| **Roadmap** | 路徑圖 | 旅遊規劃表，但可以臨時改 |
| **Stakeholder** | 有關係的人 | 婚禮要請的親戚清單 |
| **Benchmark** | 標竿對照 | 同班第一名考幾分 |

**硬規則**：
1. 每份報告首次出現英文術語 → 必帶括號中文翻譯
2. 中文翻譯後可附 1 句生活化類比（非必需，但 Deep 層建議加）
3. 縮寫（NPS / CAC / LTV）**首次必全拼出**
4. **禁止**：全篇英文術語混雜、只給中文不給英文（有些讀者需對照國際資料）

## 八、Hypothesis Transparency · 最小必備欄位

每份報告結論處必須有「📐 本報告的計算方法」抽屜，含以下 **5 個 minimum fields**：

```yaml
資料來源:
  - 類型: [DataForSEO API / 用戶作答 / 公開爬蟲 / AI 推估 / AI 生成]
  - 樣本量: N=____
  - 取樣時間: YYYY-MM-DD ~ YYYY-MM-DD
  - 代表性: [已校正地區分佈 / 未校正 / 偏 ____]
核心假設:
  - 假設 1: ____
  - 假設 2: ____
  - 假設 3: ____（最多 5 項，不少於 2 項）
信心度:
  - 數值: __%（不接受純「高/中/低」）
  - 計算方式: [基於 N 份資料點一致性 / bootstrap 取樣 / AI 自評]
失效條件:
  - 若 ____ 改變 → 本結論不成立
  - 若 ____ 改變 → 需重跑分析
建議複核時機:
  - ____ 個月後
  - 或當 ____ 事件發生時
```

**Confidence Calibration 必用 %**（不是高/中/低 · 對標 Y Combinator RFS）：
- ≥ 85% = 強結論（對外可用）
- 60-84% = 中結論（對內決策可用，對外需加 disclaimer）
- < 60% = 弱結論（標為「假設性推論」· 鮮紅 badge 提示）

## 九、差異化敘事 · BeyondPath vs Maze/Dovetail 辨識要素

**目標**：讀者遮住 logo 也能認出這是 BeyondPath 報告。

| 要素 | Maze 調性 | Dovetail 調性 | **BeyondPath 調性（本規範）** |
|------|----------|-------------|-----------------------------|
| **開場** | 一句數字 punchline | 一句 user quote | **一句賦能提問 + 一句數字判讀**（例：「這個方向值得做嗎？訊號強，但 niche 窄一半。」） |
| **語氣** | 專業冷調 · 第三人稱 | 溫暖但中性 | **溫暖精準 · 第二人稱**（稱讀者「你」） |
| **比喻來源** | 美式企業 · 無生活化 | 中性無地域 | **台灣生活化**（夜市、捷運、手搖飲、熟客） |
| **結尾** | Export / Share 按鈕 | Related insights | **「今天可以做一件事」+ 跨工具導引** |
| **視覺 DNA** | 綠色 + 白底 | 藍色 + 柔和 | **#7C5CFC 紫 + warm-serif（儀式區塊）+ 封閉五色（工具區塊）** |
| **字體系統** | Inter sans-serif | Inter sans-serif | **工具頁 sans-serif / 首頁與報告 Hero 可 Georgia italic** |
| **情緒定位** | 效率 | 共感 | **陪伴感**（「你的生意不該只靠你一個人撐」的延伸） |

**辨識度硬指標**（報告任一抽樣頁需滿足 3 項以上）：
1. 至少 1 個台灣生活化比喻
2. 至少 1 次「你」開頭的句子（不用「用戶」「使用者」稱呼讀者）
3. 至少 1 次跨工具導引（「→ 跑 Lab」「→ 走問卷」）
4. 結尾「今天可以做一件事」CTA
5. 資料源三色 badge 可見
6. 封閉五色 palette（或首頁 warm-serif + 琥珀）

## 十、驗收流程（給 Gate 3 霍爾用）

### Step 1 · 三層自查（由 prompt 或 render 執行者）

每份報告產出後，在內部 JSON 附：

```json
{
  "self_assessment": {
    "surface_score": 9.5,
    "mid_score": 9.0,
    "deep_score": 9.5,
    "avg": 9.33,
    "weakest_layer": "mid",
    "weakest_reason": "相鄰市場對比缺席",
    "auto_retry_threshold": 9.0
  }
}
```

若 avg < 9.0 → **自動重跑 prompt 一次**（Claude 自我修正）。
若二次仍 < 9.0 → **退回人工 review**（不直接交付）。

### Step 2 · Gate 3 霍爾複評（人工 · 每 10 份抽 3 份）

| 檢查項 | PASS 門檻 |
|--------|---------|
| 三層自查 avg | ≥ 9.5 |
| 術語翻譯合規 | 100% 首次出現術語有翻譯 |
| Hypothesis 5 fields | 全備 |
| Next Action 格式 | ≥ 3 條 · 三要素齊 |
| 差異化敘事硬指標 | ≥ 3 項 |
| 憲法紅線 | 零 Math.random · 零違憲色 · 零 emoji icon |

**任一 FAIL → 退版**，不接受「下一版再改」。

### Step 3 · Top-Player Benchmark（每季一次）

對照項 · 5 份報告各跑一次：

| 對標項 | 目標 % |
|--------|-------|
| Maze one-liner + next action 完備度 | ≥ 90% |
| Dovetail quote-first + cluster 完備度 | ≥ 85% |
| Productboard top-3-this-week 完備度 | ≥ 90% |
| Y Combinator RFS hypothesis 透明度 | ≥ 90% |

- **三層 × 5 報告** = **15 格**

## 十二、霍爾發現的 6 個 Edward 想法清單漏掉維度

| # | 漏掉的維度 | 重要性 | 建議 |
|---|----------|-------|------|
| **1** | **對外 vs 對內敘事差異** | 🔴 高 | 已納入市場報告 §5.3。但其他 4 份報告也該有——PATH 診斷給投資人看 vs 給自己看，內容 emphasis 不一樣。**v1.1 擴充** |
| **2** | **報告分享鏈的隱私/合規風險** | 🔴 高 | 報告含 quote 直引 → 分享外流 = GDPR/個資法風險。應有「對外版自動遮蔽 PII」開關 · 由 `suliman` 信任守護層 + 本 rubric 新增「分享敏感度分級」 |
| **3** | **報告有效期 / 時效提醒** | 🟡 中 | 市場探測 3 個月後數據可能失真。hero 標「shelf-life：N 月」+ 過期自動灰階 |
| **4** | **讀者角色自適應**（一份報告多視角） | 🟡 中 | 同一份 PATH，創辦人看結論、行銷看 Audience 章、工程看 Execution 章。切換角色 → 重新 highlight |
| **5** | **協作註解/回饋迴路** | 🟢 低（v2） | 單人團隊為主可延後，5-20 人團隊需要 |
| **6** | **歷史比較**（這次報告 vs 上次報告） | 🟢 低（v1.1） | 重跑 PATH 時對比上次分數差（「P +3 / A -2」）· 對標 Maze longitudinal study |

**霍爾觀點**：#1 #2 是**立刻該補**的缺口——#1 關乎報告的商業可用性（對外才能拿去募資 / 提案）、#2 因為是**法律風險**。建議 v1.1 優先補。
