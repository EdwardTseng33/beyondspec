## 二、G 類 · 共用驗收（10 項 × 5 報告 = 50 格）

每份報告都要通過這 10 項。任一項 < 9.5 即退版。

| # | 驗收項 | 9.5 分 criteria | 驗收方式 |
|---|--------|---------------|---------|
| G1 | **Hero verdict 3 秒測試** | 任一 persona 打開首屏 3 秒內能說出結論 | 實測 P1-P5 |
| G2 | **Next Action 3 件明確** | 每條含「誰做 / 工時 / 成功指標」三要素 | 逐條檢查 |
| G3 | **資料源 badge 100% 覆蓋** | 所有數字有 🟢/🟡/🔴 標記 | grep 數字旁無 badge 的數量 = 0 |
| G4 | **Confidence % 顯示** | 不用高/中/低，用 % 數字 | grep 「高信心 / 中信心 / 低信心」= 0 |
| G5 | **術語翻譯 100% 覆蓋** | 每個英文 buzzword 首次出現附中文（見霍爾 §7 表）| 清單逐項 |
| G6 | **空狀態有插畫 + 微文案** | 無「No data」冷敘述 | 所有 empty state 走查 |
| G7 | **獨立 shareable URL** | 每份報告有 `/path/report/{type}/{id}` 可直接打開 | URL 可貼到外部打開成功 |
| G8 | **Dark mode 適配** | `[data-theme="dark"]` 渲染不崩 | 切 dark mode 截圖 |
| G9 | **Mobile 375px 不橫滑** | 所有元素 responsive | 375 寬度截圖無橫 scroll |
| G10 | **尾部分享/儲存/下載 trio** | 三件套位置一致 | 5 報告對比 |

## 三、C 類 · P3 坤書專項（信任殺手 8 格）

P3 坤書打開 DevTools 會檢查的 8 項，任一 FAIL 即退版：

| # | 檢查項 | 9.5 分 |
|---|--------|-------|
| C1 | 無 Math.random | 全站 grep `Math\.random` 只在 UI 動畫出現，不在資料生成 |
| C2 | hash-based 推估標「模擬」 | 每個 hash 推估數字旁有 🟡 badge 或 tooltip |
| C3 | AI 生成標「非真實」 | Lab persona / AI-generated text 有 🔴 badge + 警告條 |
| C4 | API source 可查 | 每個 API 資料點可點開看 endpoint + 取樣時間 |
| C5 | 方法論 transparency | 每份報告有「📐 本報告的計算方法」抽屜 |
| C6 | reload 結果一致 | 同一 input 兩次 reload 結果相同（無隨機漂移）|
| C7 | console 0 error | DevTools console 無 error/warning |
| C8 | schema 驗證可驗 | AI 輸出 JSON 可從 network tab 看到符合 schema |

## 五、F 類 · 跨報告流暢度（6 格 · 本次新發現）

**重要發現**：前三 agent（視覺/內容/技術）都**沒覆蓋**這個維度。5 份報告目前各自孤立無引導。

**量化影響**：
- 沒有跨報告引導 → 多模組使用率 < 20%
- 有引導閉環 → 目標 50%+，留存率顯著提升

**6 條必做跨工具 CTA**：

| # | 觸發條件 | 導引去哪 | 文案 | 優先 |
|---|---------|---------|-----|------|
| F1 | PATH 診斷 A 軸（受眾）< 60 分 | → Lab 研究 | 「你的 Audience 還不夠清楚，建議跑 Lab 合成 5 位 persona 驗證」 | 🔴 P0 |
| F2 | PATH 診斷 T 軸（牽引力）< 60 分 | → 市場探測 | 「你的 Traction 弱，先看市場訊號夠不夠強」 | 🔴 P0 |
| F3 | Lab 結束 · persona 清晰 | → 問卷引擎 | 「用這份 persona 生成專屬問卷，去驗證真實用戶」 | 🟡 P1 |
| F4 | 市場探測 · 訊號強 | → 市場報告 | 「訊號足以支撐完整報告，給投資人看」 | 🟡 P1 |
| F5 | 問卷收滿 N 份 | → Lab 分析 | 「用真實回覆取代 AI 合成，產 Lab 2.0 報告」 | 🟡 P1 |
| F6 | 市場報告完成 | → PATH 重跑 | 「市場資料更新後，重跑 PATH 看分數變化」 | 🟢 P2 |

## 七、Blocker 嚴重度定義

| 嚴重度 | 定義 | 處理 |
|--------|------|------|
| **Blocker** | 用戶無法繼續，或信任崩潰（P3 Math.random）| 退版，修完重跑 Gate 全程 |

## 九、蕪菁頭發現清單漏點（2 項）

**漏點 1：跨報告流暢度完全被忽略**
前三份 agent（視覺/內容/技術）都**沒**覆蓋「用戶跑完 A 報告該去 B 報告」的引導閉環。5 報告目前各自孤立，用戶只會用第一個進來的工具就離開。**F 類 6 條跨工具 CTA 是本次最重要補充**，其中 F1-F2 由 PATH 觸發，是最多用戶的第一個接觸點。

**漏點 2：獨立 shareable URL 未納入規格**
愛德華後續明確指令「報告的連結應該也要不同且獨立的，不需要跳轉分頁來看」——G7 納入。每份報告要有 `/path/report/{type}/{id}` 獨立 URL，可 copy/share，不需要先登入進 app 再點連結跳頁才看到。

## 十、走查操作手冊

### 如何執行 25 格情緒走查

1. 每份報告建立假帳號 + 假資料集（不用真實數據）
2. 扮演每個 persona：「我是阿敏，第一次用，想知道方向對不對」
3. 記錄每時點（打開前/首屏/中段/結尾）的實際情緒反應
4. 對照 target，低 → 記錄問題 + 嚴重度

### 驗收順序
1. 先跑 C 類 P3 8 格（信任殺手優先）
2. 再跑 G 類 50 格（共用基礎）
3. 然後 P 類 30 格（報告專屬）
4. 最後 F 類 6 格（跨報告流暢）+ T 類 5 格（分享）
5. 全部 ≥ 9.5 + 一票否決項全通過 → PASS
