## 0. 一句話總結

我們選綠界托管結帳頁（app 不碰卡號）＋ 綠界電子發票 API，是把「金流 PCI」「發票上傳」兩個最重的合規負擔外包給綠界——這是對的方向，大幅降低自建風險。**但「外包金流不等於零責任」**：webhook 驗章、訂閱取消、發票折讓退費、個資告知與刪除，這四塊仍是我們自己的法律與安全責任，必須在城堡 code 裡做好。本檔把這四塊守乾淨。

## 1. 綠界信用卡（PCI）— app 不存卡號之後，仍要守什麼

### 1.1 我們選對了路：托管結帳頁 = SAQ-A 等級

- 流程：app 內選方案 → **redirect 到綠界結帳頁**（卡號只在綠界域名輸入）→ 授權成功 → 綠界 webhook 通知 app 解鎖。
- 因為 app **完全不接觸、不傳輸、不儲存卡號（PAN）/ CVV / 到期日**，PCI DSS 責任落在「SAQ-A」這個最輕的自評等級——絕大多數合規負擔由綠界承擔。**這是把卡號風險外包出去的正確設計，不要為任何理由（例如「想自己做更順的結帳 UX」）把卡號拉回 app 自存。**
- ⚠️ 綠界文件明載：使用信用卡授權交易，商家須提供 **PCI DSS 合規證明（AOC）**，由綠界依商家實際情況評估、綠界有最終審核權。代表：開定期定額前，Edward 申請綠界商店帳號時要走完綠界的商家審核（多數中小商家由綠界托管頁 = 直接符合，不需自行取得完整 PCI 認證；但這步要確認綠界核准）。[來源](https://developers.ecpay.com.tw/2868/)

### 1.2 托管頁前提下，「app 不存卡號」仍要確保的 6 件事（🔴 高）

即使卡號不在我們手上，以下若沒做好，依然會出資安事故：

1. **結帳 redirect 必須整段 HTTPS（TLS 1.2+）**，且 redirect 的目標 URL 在 code 裡寫死綠界正式域名常數、不可由前端任意參數拼裝（防 open-redirect 把用戶導到釣魚假結帳頁）。
2. **HashKey / HashIV（綠界金鑰）= 最高機密**，只能存在 server 端環境變數（城堡：Hub `.env` / GitHub Secrets），**絕不可出現在 app.html / 前端 JS / git repo / .md 任何地方**。前端只能拿到「已由 server 簽好章的表單」，拿不到金鑰本身。
3. **CheckMacValue 驗章雙向都要做**：送出綠界的表單由 server 簽 CheckMacValue；收到綠界 webhook 時**必須重算 CheckMacValue 比對**，比對失敗一律丟棄。綠界官方明列 CheckMacValue 常見錯誤（HashKey/IV 錯、參數排序錯、編碼亂碼），驗章是防偽造扣款通知的唯一防線。[來源](https://developers.ecpay.com.tw/15162/)
4. **金額在 server 端決定、不信前端**：用戶選方案後，「要扣多少錢（PeriodAmount）」由 server 依方案表算出後簽章，**前端不可傳金額由我們照單全收**（否則有人改 499→1 也能授權）。
5. **訂單編號（MerchantTradeNo）防重放**：每筆唯一、server 產生；webhook 進來先查「這張訂單是否已處理」，已處理就回 `1|OK` 但不重複解鎖（綠界同一期可能重送通知）。
6. **解鎖動作 server 端做、不靠前端**：方案解鎖、AI 額度發放，一律由收到合法 webhook 的 server 改資料庫狀態；前端只是顯示結果。**絕不可「前端拿到綠界 returnURL 就自行解鎖」**（returnURL 可被偽造，PeriodReturnURL server POST + 驗章才是真相）。[來源](https://ithelp.ithome.com.tw/questions/10195171)

### 1.3 定期定額（訂閱）授權的安全注意點（🟠 中）

- **首次 vs 後續通知走不同 URL**：首次授權結果回 `ReturnURL`，第 2 次起由綠界排程扣款、結果送 `PeriodReturnURL`。**兩個 URL 都要驗章、都要 idempotent。** `PeriodReturnURL` 每期只送一次，沒收到要主動用「定期定額訂單查詢 API」補查實際扣款結果（不可假設沒收到=沒扣）。[來源](https://www.ecpay.com.tw/CascadeFAQ/CascadeFAQ_Qa?nID=1202)
- **連續扣款失敗 6 次 → 綠界自動取消後續扣款**。我們 code 要監聽失敗通知：失敗時 → 寄信通知用戶更新卡片 + app 標記「付款異常」寬限狀態（不是立刻斷服務，給補卡時間），達自動取消門檻後才降級。[來源](https://www.ecpay.com.tw/Announcement/DetailAnnouncement?nID=4113)
- **訂閱取消要真的停扣**：用戶在 app 內按「取消訂閱」，我們**必須呼叫綠界「信用卡定期定額訂單作業 API」實際停用該筆授權**——不能只在 app 自己資料庫標記取消、卻沒去綠界停扣，否則用戶下個月仍被扣款 = 消費爭議 + 個資/金流信任崩。這是 🔴 高優先必做。[來源](https://support.ecpay.com.tw/16214/)
- **Team 改人改金額**：Team owner 加/減 seat → 總額變 → 綠界定期定額**金額不能線上直接改**（定期定額一旦建立，金額固定）。實務做法：停用舊授權 → 用新總額建新授權（請用戶重新確認一次扣款金額），或下一帳期生效。這個 UX 要設計清楚，避免「以為改了金額其實沒改」。

## 2. 電子發票 — 台灣財政部規範重點

> 我們用綠界電子發票 API，扣款成功自動開立。綠界負責「每日排程上傳財政部平台」，但**字軌申請、開立內容正確性、折讓/作廢/退費的合法操作，是我們的責任**。

### 2.1 開立後上傳時限（綠界代傳，但我們要知道）（🟠 中）

- **B2C（個人戶，二聯式）**：開立後 **48 小時內**上傳財政部電子發票整合服務平台。
- **B2B（公司戶，三聯式，買受人為營業人）**：買受人至遲應於開立後 **7 日內**完成接收，開立人同時段內傳輸存證。
- 實務上由綠界工具「每日排程自動上傳」，不需人工——但代表：**開錯/漏開不能拖**，發票一旦上傳就進財政部，更正要走作廢/折讓流程（見 2.4），比開立前改麻煩得多。[來源](https://a1.digiwin.com/essay/essay-content.php?detail=107)·[財政部 NEW 須知](https://www.ntbsa.gov.tw/download/933a0a948fc34844befe9e729e10826d)

### 2.2 字軌配號（🔴 高 — Edward 申辦前置）

- 發票字軌 = **2 個英文字母 + 8 位數字**，用以辨識追蹤發票。
- 必須先到**公司登記地的稽徵單位（國稅局）申請發票字軌**，取得配號後才能開立。**字軌按期（每 2 個月一期，奇數月）配號**，要預估用量足夠（訂閱制每月開票，量穩定，但成長要記得加配）。
- B2B 公司字軌需另外申請。實務上多數 SaaS 透過**加值中心（綠界即為財政部核准的加值服務中心）**代為處理字軌與配號，降低自辦複雜度——這條 Edward 開綠界帳號時一併確認綠界幫忙配號。[來源](https://www.ezreceipt.cc/blog/view/8353)·[B2B 字軌申請](https://www.ezreceipt.cc/blog/view/7400)

### 2.3 三聯式（統編）vs 二聯式（手機載具）— 我們 code 要分流（🟠 中）

| 客戶型態 | 發票類型 | 必填欄位 | 我們 app 收什麼 |
|---|---|---|---|
| **公司戶（多數 B2B 客戶）** | 三聯式 | **統一編號（8 碼）** + 公司抬頭 | 結帳時要欄位：統編 + 抬頭，統編做 8 碼格式驗證 |
| **個人戶** | 二聯式 | 手機條碼載具 / 自然人憑證 / 捐贈碼 | 結帳時要欄位：載具類型 + 載具號碼（手機條碼格式 `/` 開頭 8 碼）|

- **統編格式驗證**：8 位數字 + 財政部統編檢核碼演算法，前端先驗、server 再驗，避免開錯抬頭發票（開錯要作廢重開，很麻煩）。
- **手機載具格式驗證**：`/` + 7 碼英數，避免載具歸戶失敗（用戶領不到發票會客訴）。
- **Team**：發票開給 owner（公司戶填統編 + 抬頭），一張總額發票。

### 2.4 折讓 / 作廢 / 退費（退訂 / 降級退費時必走）（🔴 高）

這是訂閱制最容易踩的合規坑——用戶退費 / 中途降級退款時：

- **作廢**：綠界系統**僅接受單月 13 日前作廢上期發票**；**每年奇數月的 13 號 23:59:59 之後，因已申報財政部，無法作廢前兩個月的發票**。→ 代表：跨月退費往往**不能作廢、只能開折讓**。
- **折讓**：折讓**無期限限制**，應於「折讓事實發生時」開立折讓證明單。退費 / 部分退款一律走折讓最安全。
- **順序鐵則**：若該發票已有折讓紀錄，**不得直接作廢**——要先把全部折讓紀錄作廢，才能作廢該發票。我們 code 的退費流程要照這個順序，否則 API 會擋。
- **退費 = 同時要做三件事**：① 綠界停用定期定額授權（停扣）② 開折讓或作廢（稅務）③ app 內降級/鎖定（產品）。**三件缺一都會出爭議**——這是退費 SOP 必須三件綁一起。[來源](https://support.ecpay.com.tw/6897/)·[作廢折讓 API](https://developers.ecpay.com.tw/?p=7911)

## 3. 個資法（台灣 PDPA）— 付款 + 帳號資料

> 我們是「非公務機關」，蒐集用戶帳號 + 付款相關個資，受個人資料保護法規範。核心三件：蒐集要告知、用途不可超目的、目的消失要刪。

### 3.1 蒐集告知（§5 / §8）（🔴 高 — 上線前必備）

- **§8 告知義務**：蒐集個資時應向當事人告知：① 蒐集者（BeyondSpec / Edward 的營業主體名稱）② 蒐集目的（提供 SaaS 服務、計費、開立發票、客服）③ 個資類別（姓名、email、統編/抬頭、付款狀態紀錄——**注意：卡號在綠界、不在我們，告知時要講清楚「金流由綠界處理，本服務不儲存您的信用卡卡號」**）④ 利用期間、地區、對象、方式 ⑤ 當事人可行使的權利（查詢/更正/刪除/停止利用）⑥ 不提供的後果。
- **§5 比例原則**：只蒐集「提供服務 + 計費 + 開發票」必要的資料，不超蒐。Starter 試用**不需信用卡 = 符合資料最小化**（這個設計很好，繼續保持）。
- **落地方式**：上線前必備**隱私權政策 + 服務條款**頁面，註冊/結帳前用 checkbox「我已閱讀並同意」(不可預設打勾)。這是 🔴 高、收真錢前硬需求。[個資法全文](https://law.moj.gov.tw/LawClass/LawAll.aspx?PCode=I0050021)

### 3.2 留存與刪除（§11）— Starter 鎖定後 90 天清資料的合法性（🟠 中 → 合法，但要補一步）

主依據設計：Starter 7 天未付 → 軟鎖（資料保留）→ 90 天仍未付 → 寄信預告 → 清資料。**從 PDPA 看這個設計是合法且合規友善的**，理由：

- **§11**：特定目的消失或期限屆滿時，應主動或依當事人請求**刪除、停止處理或利用**該個資。「但因執行職務或業務所必須**或經當事人書面同意**者，不在此限。」
- Starter 用戶不付費 = 服務關係（特定目的）走向終結，90 天後清資料正是履行 §11 的「主動刪除」義務——**不無限留存反而是守法**，比「永遠留著」更安全。
- ⚠️ **但要補一步合法性**：清資料前的「寄信預告」要明確、要留存發送紀錄（證明有通知）。且隱私權政策裡要**事先寫明這個保留/刪除政策**（「免費帳號停用後保留 90 天，逾期未升級將刪除您的資料」），讓刪除是「事先告知過的政策」而非突襲。這樣完全站得住。

### 3.3 發票/交易資料的法定保存期限 — 不能跟 90 天混為一談（🔴 高 — 容易踩的坑）

**這是最容易出錯的點，務必區分兩種資料：**

| 資料種類 | 保留政策 | 依據 |
|---|---|---|
| **產品使用資料**（看板/客戶/專案內容、AI 對話） | Starter 鎖定 90 天後可清 | §11 特定目的消失 |
| **交易 / 發票 / 會計憑證**（已扣款紀錄、已開發票） | **必須依稅法保存（帳簿憑證至少 5 年、營業稅相關 7 年）— 不可因用戶要求刪除或 90 天到期就刪** | §11 但書「業務所必須 + 有法令規定保存期限」+ 稅捐稽徵法 / 商業會計法 |

- 代表：用戶就算行使「刪除權」或 Starter 90 天清資料，**已開立的發票、已扣款的交易紀錄要保留（法令保存期）**——這正是 §11 但書「因執行業務所必須、有法令規定保存期限」的例外。
- **code 設計**：清資料時要能「刪產品內容、保留財會憑證」分離（不是一鍵刪全部）。隱私政策也要寫明「依稅法規定，交易與發票紀錄將保存法定年限」。

### 3.4 Team 成員資料 — 誰是蒐集者？（🟠 中 — 釐清責任歸屬）

Team owner 邀請成員加入，成員的姓名/email 進到我們系統：

- **雙層蒐集者關係**：① BeyondSpec 對「owner」是直接蒐集者（owner 自己註冊、同意條款）。② 對「被邀請的成員」，**owner 是該成員個資的提供來源 + 共同蒐集者**，BeyondSpec 是受 owner 委託處理。
- **責任設計**：
  - owner 邀請成員時，**邀請流程要讓 owner 確認「我已取得成員同意提供其資料」**（一個 checkbox / 條款），把「向成員告知」的第一線責任放在 owner（他才是有勞雇/合作關係的人）。
  - 成員首次登入時，**BeyondSpec 仍要向成員顯示自己的隱私政策**並取得成員同意（成員直接面對我們系統，我們有自己的告知義務，不能完全靠 owner 轉達）。
  - 成員被移出 Team → 該成員在此空間的個人識別資料應可移除（成員的「刪除權」對應）。
- **白話**：owner 負責「我有權拉這個人進來」，BeyondSpec 負責「這個人進來後我有自己對他的告知與權利保障」。兩層都要有，責任才清楚。

### 3.5 第三方 vendor 風險（🟢 低 — 記錄備查）

接真錢後我們的個資/金流會經過：

| Vendor | 拿到什麼 | 我們要確認 |
|---|---|---|
| **綠界 ECPay** | 卡號（他存，我們不存）、扣款金額、發票買受人資料 | 已是財政部核准加值中心 + 金管會許可第三方支付；簽約即有其合規承諾。簽約時保留其 DPA / 隱私條款備查 |
| **Firebase（Google）** | 帳號、email、產品資料 | Google 有 SOC2/ISO27001 + GDPR DPA；資料落地區域確認（建議鎖亞洲區） |
| **Anthropic（Claude）** | AI 功能的對話內容 | 確認商用條款不拿用戶資料訓練 |

接第一個企業客戶要 SOC 2 / DPA 時，再升級正式 vendor 風險評估（套城堡 Vanta pattern）。現階段記錄即可。

## 4. 收真錢前 Gate 5 信任 checklist

> 上線「開始收費」前逐項過。每項要附證據（截圖 / code 位置 / 設定畫面），不接受空口 PASS。
> 提醒（6/5 定位）：以下是「收真錢前我建議補齊的保護」+ 我會主動把能寫的鎖/驗章/格式驗證寫好。最終上不上線 Edward 拍板。

### A. 金流安全（綠界）
- [ ] 綠界商店帳號已開、商家審核通過（含定期定額 + 電子發票權限開通）
- [ ] HashKey / HashIV 只在 server 環境變數，grep 全 repo（含 .md / .sh / .js / app.html）無金鑰外洩
- [ ] 結帳 redirect 目標域名 = code 寫死綠界正式域名常數，無 open-redirect
- [ ] webhook（ReturnURL + PeriodReturnURL）**雙向 CheckMacValue 驗章**已實作，驗章失敗丟棄
- [ ] 扣款金額（PeriodAmount）server 端依方案表計算後簽章，前端不可竄改
- [ ] 訂單編號唯一 + webhook idempotent（同期重送不重複解鎖）
- [ ] 方案解鎖 / AI 額度發放只由合法 webhook 觸發，前端不自行解鎖
- [ ] 「取消訂閱」實際呼叫綠界停用定期定額 API（驗證綠界後台確實停扣）
- [ ] 扣款失敗 6 次自動取消的監聽 + 寬限期 UX 已做

### B. 電子發票
- [ ] 發票字軌已申請、配號足夠（綠界代配已確認）

## 5. 三條紅線（絕不能在未滿足時上線收費）

> 6/5 定位下，這三條是我能給的**最強建議 + 我已把能做的保護做好**——不是否決權。但這三條若沒做就收真錢，幾乎必然出消費爭議或違法事故，強烈建議 Edward 不要在缺這三條時開收費。

### 🔴 紅線 1 · webhook 不驗章 / 前端自行解鎖 → 絕不上線
若方案解鎖、付款確認靠「前端拿到 returnURL 就解鎖」而非「server 收 PeriodReturnURL + CheckMacValue 驗章」，等於任何人偽造一個成功通知就能白嫖或冒充付款。**金流真相只認 server-side 驗章後的通知。** 沒做 = 不收費。

### 🔴 紅線 2 · 取消訂閱沒有真正停扣綠界授權 → 絕不上線
用戶按取消、我們只在自家資料庫標記、卻沒呼叫綠界停用定期定額 API → 用戶下月繼續被扣 → 信用卡爭議 + 個資/金流信任徹底崩 + 可能觸及消保。**「取消」必須等於綠界後台確實停扣**，且要有驗證證據。沒做 = 不收費。

### 🔴 紅線 3 · 沒有隱私政策/服務條款 + 同意機制，或交易紀錄被當一般資料刪 → 絕不上線
收費即蒐集付款個資，無 §8 告知頁面 + 無「不可預設打勾」的同意 checkbox = 違反個資法。**且最容易致命的坑**：把「已開發票/已扣款的交易憑證」跟「產品使用資料」一起在 90 天/刪除權時清掉 → 違反稅法法定保存（帳簿憑證 5 年、營業稅 7 年）。交易財會憑證必須與產品資料分離保存。沒做 = 不收費。

| 項目 | 為什麼只有 Edward 能做 | 前置依賴 |
|---|---|---|
| **綠界 ECPay 商店帳號（特約商店）** | 需營業主體資料 + 負責人身分 + 銀行帳戶，金流契約 | 收費功能的總前置；含定期定額 + 電子發票權限開通 |
